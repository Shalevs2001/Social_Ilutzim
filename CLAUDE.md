## Architecture

### State split: localStorage vs Firebase

**localStorage** (admin-only, single machine):
- `ks_schedule` Рђћ the weekly schedule
- `ks_employees` Рђћ employee roster with quotas
- `ks_settings_v5` Рђћ mandatory requirements
- `ks_shift_priority` Рђћ auto-schedule priority order
- `ks_schedule_date`, `ks_schedule_notes` Рђћ metadata shown on export

**Firebase Realtime DB** (synced across admin + employee devices):
- `availability/{empId}/{dayKey}/{shiftType}` Рђћ employee availability
- `submissions/{empId}` Рђћ submission status per employee
- `approvalRequests/{empId}` Рђћ pending manager-approval requests
- `notifications/{id}` Рђћ admin notification feed
- `shiftTimes/{type}` Рђћ editable shift hours (visible in employee view)
- `weekendMiddleActive/{fri|sat}` Рђћ whether the optional weekend-middle shift is active
- `employeeNotes/{empId}/{noteId}` Рђћ per-employee checklist notes
- `availabilitySnapshots/{empId}`, `notesSnapshots/{empId}` Рђћ saved at submission, used for restore

### AppContext (`src/context/AppContext.jsx`)

Single context that owns all state and actions. Both pages use `AppProvider` (the employee page gets its own instance). Key concept: **draft mode** Рђћ when an employee is filling in their availability, changes are local (`availDraft`) and only flushed to Firebase on submit (`_flushDraft`). `mergedAvailability` is what all consumers read: Firebase data for everyone except the current drafter who sees their local draft.

Always interact with `AppContext` through the exported `useApp()` hook.

### Schedule data shape

```js
schedule[dayKey] = {
  slots: [{ id, type, employee, employee2, status, note, reshemBetMark, konenutMark, time, label }],
  adHocShifts: [...],   // same shape Рђћ added dynamically
  reshemBetBackup: bool,
  weekendMiddle: bool,
}
```

`slots` are the fixed shifts generated by `makeEmptySchedule()`. `adHocShifts` are added by the admin via AdHocModal or toggles (reshem-bet, weekend-middle).

### Shift types (`src/constants.js`)

Weekday: `morning`, `short_morning`, `middle`, `evening`, `samples`  
Weekend: `weekend_morning`, `weekend_middle`, `weekend_evening`  
Special: `custom` (ad-hoc, any day), `reshem_bet` (backup broadcast)

`short_morning` and `samples` count as **0.5 shifts** toward quota (`shiftWeight()`).

`SLOT_ALTERNATIVES`: `short_morning` slot accepts `morning` availability; `samples` slot accepts `evening` availability.

### Employee shape

```js
{ id, name, quota, minQuota, joker, gender, preferences }
```

`joker` employees are exempt from quota counting. `minQuota` triggers a score penalty in auto-schedule if not reached. `preferences` contains pairing rules, shift-type preferences, and clopen-avoidance preferences (see `DEFAULT_PREFERENCES` in constants).

### Auto-schedule (`src/utils/autoSchedule.js`)

Score-based, two-pass (regular availability first, then low-priority if under quota). Key factors: quota headroom, minQuota gap penalty, clopen prevention (evening Рєњ next morning), pairing preferences, notAlone rule. Processes slots in priority order from `shiftPriority`.

### PNG Export (`src/utils/exportSchedule.js`)

Pure Canvas 2D drawing Рђћ no DOM capture. Chosen because html2canvas/html-to-image both had RTL text rendering issues. `scheduleNotes` may contain rich-text HTML (bold/highlights from the notes editor); `stripHtml()` converts it to plain text before canvas rendering.

### Settings versioning

`SETTINGS_VERSION = 5` in constants. A version mismatch wipes `ks_settings` and resets to defaults (migration runs at module load in AppContext before any hooks fire).

### Auto-deploy via GitHub Actions

Every push triggers a build on GitHub's servers and deploys via rsync over SSH.

- `.github/workflows/deploy.yml` Рђћ runs on push to `main` Рєњ deploys to port 3000
- `.github/workflows/deploy-staging.yml` Рђћ runs on push to `dev` Рєњ deploys to port 3001

Required GitHub Secrets (Settings Рєњ Secrets Рєњ Actions):
- `HETZNER_HOST` Рђћ server IP
- `HETZNER_USER` Рђћ SSH username (root)
- `HETZNER_SSH_KEY` Рђћ private SSH key for deployment

2. Make changes, then commit and push:

3. Check the change works on **http://46.224.228.163:3001**

   ```bash

   git checkout dev
   ```

Never commit directly to `main`. If Claude Code suggests a push without first checking out `dev`, correct it.
