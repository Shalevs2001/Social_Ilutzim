## What this project is

Weekly shift scheduler for the "סושיאל כאן חדשות" news outlet. Two separate pages:
- `/` — admin view: build the schedule, view employee availability, manage settings
- `/employee` — employee view: submit weekly availability constraints

Stack: React 18 + Vite + Tailwind CSS (RTL/Hebrew throughout) + Firebase Realtime Database.

## Architecture

### State split: localStorage vs Firebase

**localStorage** (admin-only, single machine):
- `ks_schedule` — the weekly schedule
- `ks_employees` — employee roster with quotas
- `ks_settings_v5` — mandatory requirements
- `ks_shift_priority` — auto-schedule priority order
- `ks_schedule_date`, `ks_schedule_notes` — metadata shown on export

**Firebase Realtime DB** (synced across admin + employee devices):
- `availability/{empId}/{dayKey}/{shiftType}` — employee availability
- `submissions/{empId}` — submission status per employee
- `approvalRequests/{empId}` — pending manager-approval requests
- `notifications/{id}` — admin notification feed
- `shiftTimes/{type}` — editable shift hours (visible in employee view)
- `weekendMiddleActive/{fri|sat}` — whether the optional weekend-middle shift is active
- `employeeNotes/{empId}/{noteId}` — per-employee checklist notes
- `availabilitySnapshots/{empId}`, `notesSnapshots/{empId}` — saved at submission, used for restore

### AppContext (`src/context/AppContext.jsx`)

Single context that owns all state and actions. Both pages use `AppProvider` (the employee page gets its own instance). Key concept: **draft mode** — when an employee is filling in their availability, changes are local (`availDraft`) and only flushed to Firebase on submit (`_flushDraft`). `mergedAvailability` is what all consumers read: Firebase data for everyone except the current drafter who sees their local draft.

Always interact with `AppContext` through the exported `useApp()` hook.

### Schedule data shape

```js
schedule[dayKey] = {
  slots: [{ id, type, employee, employee2, status, note, reshemBetMark, konenutMark, time, label }],
  adHocShifts: [...],   // same shape — added dynamically
  reshemBetBackup: bool,
  weekendMiddle: bool,
}
```

`slots` are the fixed shifts generated by `makeEmptySchedule()`. `adHocShifts` are added by the admin via AdHocModal or toggles (reshem-bet, weekend-middle).

### Shift types (`src/constants.js`)

Weekday: `morning`, `short_morning`, `middle`, `evening`, `samples`  
Weekend: `weekend_morning`, `weekend_middle`, `weekend_evening`  
Special: `custom` (ad-hoc, any day), `reshem_bet` (backup broadcast)

`short_morning` and `samples` count as **0.5 shifts** toward quota (`shiftWeight()`).

`SLOT_ALTERNATIVES`: `short_morning` slot accepts `morning` availability; `samples` slot accepts `evening` availability.

### Employee shape

```js
{ id, name, quota, minQuota, joker, gender, preferences }
```

`joker` employees are exempt from quota counting. `minQuota` triggers a score penalty in auto-schedule if not reached. `preferences` contains pairing rules, shift-type preferences, and clopen-avoidance preferences (see `DEFAULT_PREFERENCES` in constants).

### Auto-schedule (`src/utils/autoSchedule.js`)

Score-based, two-pass (regular availability first, then low-priority if under quota). Key factors: quota headroom, minQuota gap penalty, clopen prevention (evening → next morning), pairing preferences, notAlone rule. Processes slots in priority order from `shiftPriority`.

### PNG Export (`src/utils/exportSchedule.js`)

Pure Canvas 2D drawing — no DOM capture. Chosen because html2canvas/html-to-image both had RTL text rendering issues. `scheduleNotes` may contain rich-text HTML (bold/highlights from the notes editor); `stripHtml()` converts it to plain text before canvas rendering.

### Settings versioning

`SETTINGS_VERSION = 5` in constants. A version mismatch wipes `ks_settings` and resets to defaults (migration runs at module load in AppContext before any hooks fire).

## Key component responsibilities

| Component | Role |
|---|---|
| `App.jsx` | Admin shell: header, sidebar (draggable employee cards), view switcher (schedule / availability / split) |
| `ScheduleGrid.jsx` | 7-column grid + toolbar + date/notes bar + rich-text notes editor (double-click to expand) |
| `DayColumn.jsx` | One day's column: toggle buttons (reshem-bet, weekend-middle) + slot list |
| `ShiftSlot.jsx` | Single shift card: drop target, employee chips, manual-name input, badges (reshem-bet/konenut) |
| `DndProvider.jsx` | dnd-kit context; drag employee → slot assigns, drag slot → slot swaps via `applySchedulePatches` |
| `AvailabilityGrid.jsx` | Admin + employee availability matrix; cells cycle none→regular→low on click |
| `SettingsModal.jsx` | 4 tabs: mandatory requirements, shift times, shift priority order, per-employee preferences |
| `EmployeeView.jsx` | Employee-facing page: name picker → availability form → submit/approve flow |

## Routing & deployment

```
main.jsx
  /           → App (admin)
  /employee   → EmployeeViewPage
  /view       → ScheduleViewPage (read-only, real-time)
```

All URLs are built dynamically from `window.location.origin` — no hardcoded domains anywhere.

Clipboard copy uses `src/utils/clipboard.js` which falls back to `execCommand` on HTTP (required for non-HTTPS servers).

### Live server: Hetzner (46.224.228.163)

- **Production** → port 3000 → served from `/var/www/shift-scheduler/`
- **Staging** → port 3001 → served from `/var/www/shift-scheduler-staging/`
- nginx config at `/etc/nginx/sites-available/shift-scheduler` (and `-staging`)
- Static files served by nginx with `try_files $uri /index.html` for SPA routing

### Auto-deploy via GitHub Actions

Every push triggers a build on GitHub's servers and deploys via rsync over SSH.

- `.github/workflows/deploy.yml` — runs on push to `main` → deploys to port 3000
- `.github/workflows/deploy-staging.yml` — runs on push to `dev` → deploys to port 3001

Required GitHub Secrets (Settings → Secrets → Actions):
- `HETZNER_HOST` — server IP
- `HETZNER_USER` — SSH username (root)
- `HETZNER_SSH_KEY` — private SSH key for deployment

### Branch workflow (non-technical summary)

1. Always work on the `dev` branch — changes go to staging (port 3001) for review
2. When ready to publish to production: merge `dev` into `main`
3. Never commit directly to `main` unless it's an urgent fix

```bash
git checkout dev          # switch to dev branch
# ... make changes ...
git add -A
git commit -m "description"
git push                  # → auto-deploys to staging

# When ready for production:
git checkout main
git merge dev
git push                  # → auto-deploys to production
git checkout dev          # go back to dev
```

### Firebase: sharedSchedule

The admin schedule auto-syncs to Firebase `sharedSchedule` every 1.5s. The `/view` page listens in real-time with `onValue`. Visibility is controlled by `scheduleVisible` (localStorage → Firebase).

For Netlify: `public/_redirects` contains `/* /index.html 200`.
For Vercel: `vercel.json` with rewrites to `/index.html`.

The admin view has no authentication — anyone with the URL can access it.
